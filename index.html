Motivation: Why Joker?

In today's fast-paced digital world, moments of simple joy and spontaneous laughter are becoming rare. Existing entertainment devices are often passive or require complex interactions.

Joker aims to solve this by providing an interactive, tangible, and localized humor experience. It's not just a screen; it's a smart companion that reacts to physical inputs (keypad), understands context (ranking system), and breaks language barriers (real-time translation) to deliver happiness instantly.

Core Features

Joker integrates multiple IoT capabilities to create a rich user experience:

  • Smart Retrieval: Fetches random jokes via API based on user-selected categories (Programming, Dark, Pun, etc.).
  • Real-time Translation: Automatically translates English jokes into Korean for a bilingual experience.
  • Physical Interaction: Supports 'Keypad Input' for controls.
  • Auditory & Visual Feedback: Adaptive Buzzer sounds based on ratings.
  • Ranking System: Tracks user ratings in the cloud and displays the 'Top 3 Hall of Fame'.

My Task

The core focus of my task was establishing a reliable Cloud-DB synchronization pipeline and optimizing the user interface for real-time feedback. I managed the end-to-end flow from physical input to automated server-side data processing.

1. Data Logging & Rating System

Joke Database & Rating Records in Google Sheets

I implemented an interactive rating system using a 4x4 Keypad to bridge the gap between user experience and data collection:

  • Keypad Interaction: Users can rate a joke from 1 to 5 immediately after reading by pressing the corresponding numeric key.
  • Transmission: The ESP32 captures this input and transmits the rating along with the joke's unique ID to the cloud via Webhooks.
  • Storage: Data is logged in Google Sheets, maintaining a persistent database for original jokes, Korean translations, and user-assigned ratings.

2. Automated Sorting (App Script & Trigger)

Google App Script Logic

Automation Trigger

To automate the leaderboard management, I developed a server-side sorting logic:

  • Custom Scripting: Wrote a Google App Script to monitor the rating column and sort the entire database in descending order automatically.
  • Trigger Mechanism: Set up an Event-driven Trigger (On Edit) that executes the sorting script the moment a new rating is added, ensuring the Hall of Fame is always current.

3. Simulation Constraints & Debugging

Ranking Menu (Key 'A')

Fetching Cloud Data

Technical Problem: During integration within the Wokwi Simulator, we faced persistent HTTPS handshake failures when pulling sorted ranking data to the LCD panel.

Observation: This is a simulator-specific network restriction. While the LCD displays a 'Loading' state in Wokwi, the logic is fully verified. Using physical ESP32 hardware would resolve these connection blocks, enabling a seamless Ranking display.

4. Evolution of Time Tracking Feature

V1: Elapsed Seconds

V2: Final Timestamp Display

I refined the "Last Update" feature to improve both technical stability and user readability:

  • The Pivot: Initially, I implemented a feature showing elapsed seconds (V1) since the last update. However, this caused synchronization issues with external APIs.
  • Final Logic: I transitioned to a static timestamp display (V2) showing the exact time of the last successful retrieval. This improved network stability and provided a clearer reference for the user.
